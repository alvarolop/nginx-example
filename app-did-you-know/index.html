About This Demo

This demo showcases how NGINX serves content on OpenShift, using different methods of file management and storage. Below is an overview of how three specific pages are handled.

  • Page 3: This page is not part of the original container image and is copied to the container each time the pod starts. Since it is not stored in a PVC, it will not persist across pod restarts and needs to be re-copied if the pod is redeployed.

  • Page 4: Managed via a ConfigMap, this page is mounted to the pod at startup, making it available dynamically. This method allows you to inject content or configurations without modifying the container image.

  • Page 5: Like Page 3, this page is copied during startup, but to a location backed by a PVC. This ensures that it persists even if the pod is restarted.

Feel free to explore the different pages and see how NGINX can serve content dynamically from various sources, including direct container copy, ConfigMaps, and PVCs. This setup highlights the flexibility and power of managing containerized applications in a Kubernetes environment like OpenShift.
